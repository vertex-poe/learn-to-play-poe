# Roadmap Details

This file holds the full description for every item listed in `ROADMAP.md`. `ROADMAP.md` holds only the point-form headline; this file holds the "why/how" behind each one, in the same order under the same `## Goal:` headings.

**The two files are 1:1.** Every headline in `ROADMAP.md` must have exactly one matching entry here, and vice versa. When you add, remove, or reword an item, update both files together in the same change — removing an item from one without removing it from the other leaves them out of sync.

Everything here can be considered aspirational and will likely never see the light of day. Item ideas may not be fleshed out and change drastically or be considered an actual bad idea the morning after we wrote it down in the middle of the night.

This file tracks unimplemented work only — once an item is done, remove it from both files rather than checking it off.

## Goal: poe-info-service

Work items derived from `poe-info-service/docs/decisions/` (ADR-001 through 005) and root `docs/decisions/006-poe-info-service.md`, comparing the decisions against the current implementation.

- [ ] Shared, addon-agnostic install location + bootstrap-if-newer: `ServiceManager::start()` (`src/services/ServiceManager.cpp`) currently launches `poe-info-service(.exe)` straight out of this app's own install directory; per ADR-001/ADR-002 the app must instead bootstrap its bundled copy into a shared location only if newer, then always launch the shared copy from there
- [ ] Replace spawn-tied process lifecycle: `ServiceManager` currently binds the service's life to this app via a Windows Job Object (`KILL_ON_JOB_CLOSE`) and `PR_SET_PDEATHSIG` on Linux; ADR-001 supersedes this with a keep-alive-based model — any client can start or restart the service, and it must outlive whichever addon happened to launch it
- [ ] Versioned WebSocket API: negotiate a client-facing API version (e.g. `/v1`, `/latest`) at connection handshake, separate from the existing peer singleton-election version check in `proto.go`; once shipped, a version's response shapes are permanent — fields are never removed, renamed, or repurposed (ADR-003)
- [ ] CI schema-compatibility gate: migrate a scratch DB to head and verify every still-supported API version's data-shaping logic still succeeds, as defense-in-depth for the additive-only migration discipline already assumed by the schema (ADR-003)
- [ ] Self-update mechanism: the running service periodically checks a durable release feed (e.g. GitHub Releases), verifies signature/checksum, and updates itself in place — not yet implemented anywhere in `poe-info-service` (ADR-002)
- [ ] Manual installer: standalone poe-info-service installer for troubleshooting/recovery, installs into the shared location only if what's there isn't already newer (ADR-002)
- [ ] Binary signing + checksum verification: required before any downloaded or self-installed binary is written to the shared location or executed, regardless of which of the three distribution paths delivered it (ADR-002)
- [ ] Credential storage package — macOS/Linux backends: `internal/creds` (`Store`/`Get`/`Delete`, build-tag-selected per platform) now has a Windows backend (danieljoos/wincred); still needs keybase/go-keychain (macOS) and godbus/dbus Secret Service (Linux), plus an in-memory backend for automated tests (ADR-005)
- [ ] Credential expiry/staleness policy: explicitly left open by ADR-004/ADR-005 and not yet the subject of a dedicated ADR — needs its own design pass now that the OAuth PKCE flow (`internal/poe`) is the first credential type this actually matters for
- [ ] PoE OAuth data endpoints: `internal/poe` and the `poe.oauth.*` WebSocket methods only cover authentication (login/status/logout, token refresh) plus, as of 2026-07-13, `/profile`'s `locale`/`twitch` fields (`poe.profile.locale`/`poe.profile.twitch`, `internal/server/poe_profile.go`) and `/leagues` (`poe.leagues.list`, `internal/server/poe_leagues.go`, also 2026-07-13) — the rest of the `api.pathofexile.com` data endpoints (`/character`, `/stash/{league}`, per `_reference/poe-apis/poe-apis.md` §6.2) are still not wired up. Both `/profile` and `/leagues` are fetched through the reqqueue-backed queue (see below), never pre-fetched, only fetched and cached lazily when actually requested and once the cached copy exceeds a max-age (defaults: `poeProfileLocaleCacheTTL` 30 days, `poeProfileTwitchCacheTTL` 7 days, `poeLeaguesCacheTTL` 6 hours — leagues change far more often than a profile's static fields, but still rarely enough that a request-scoped fetch is fine), each clamped to its own minimum re-fetch floor (`poeProfileMinRefetchAge` 1 hour, `poeLeaguesMinRefetchAge` 5 minutes) regardless of what a caller requests. `/leagues` is cached in its own `leagues` SQL table (unique on `name`+`realm`, since a league name like "Standard" repeats across realms) rather than the generic `api_cache` blob store `/profile` uses, since it's naturally tabular, queryable data — see `poe-info-service/docs/schema.md`. It's also the one endpoint here needing no `account` selector at all: `GET /leagues` requires no Bearer token and returns account-independent data, so `internal/poe.Client.FetchLeagues` sends no Authorization header and `ensureLeagues` never needs to resolve an active account before scheduling a fetch (unlike `ensurePoeProfile`). `poe.profile.*` still accept an optional `account` selector (a `poe_uuid` or an `accounts.name`) and an optional `priority` override (defaulting to `poeProfileLocaleFetchPriority`=High / `poeProfileTwitchFetchPriority`=Low, vs. `poeLeaguesFetchPriority`=Medium) — see the "Account selector" and request-queue entries below for how those fit the broader design. Characters/stash still need their own wiring once built, reusing the same queue instance and priority/account-selector conventions. **Fetch policy, cost reporting, and single-league detail added** (2026-07-13, following an architecture review by Opus): `ensurePoeProfile`/`ensureLeagues`/`ensureLeagueDetail` now take an explicit `fetchPolicy` (`"ifStale"`/`"never"`/`"always"`, `internal/server/poe_fetch.go`) orthogonal to `maxAge` — `"never"` is a pure read-only peek that serves a stale/missing cache without ever spending a query, letting a caller say "give me whatever's cheaply available" without risking an unwanted fetch. The response vocabulary grew two new orthogonal fields, `freshness` and `fetching`, alongside the original `status` (kept, per ADR-003, with two new values, `"stale"`/`"miss"`, added to it) — and a `"pending"`/peek response now carries whatever's cached (even stale) instead of nothing. `poe.leagues.detail` (`internal/server/poe_leagues.go`) serves one league by name, reusing `poe.leagues.list`'s exact bulk fetch and dedup key since the OAuth API has no dedicated single-league endpoint yet (see the new roadmap item on swapping in a real one if that changes). An opt-in `includeCost` request param surfaces `proto.FetchCost` (which policy, how many queries, that policy's rule state) on a response that actually fetched; `poe.ratelimit.status`/`TopicPoeRateLimit` separately expose every policy's live budget straight from `reqqueue.Queue.Policies` (added to the queue itself, see below) — kept as its own endpoint/topic rather than stapled onto every domain response, since it's shared state that would otherwise duplicate and go stale across payloads
- [ ] Swap poe.leagues.detail's bulk-refetch stand-in for a real single-league OAuth endpoint, if/when PoE ever ships one: `poe.leagues.detail` (2026-07-13) has no dedicated PoE OAuth endpoint to call — the Legacy API has one (`GET /api/leagues/{id}`, per `_reference/poe-apis/poe-apis.md` §"Leagues (expanded)"), but the OAuth API doesn't mirror it — so a "refresh" today just re-runs `poe.leagues.list`'s bulk `GET /leagues` fetch (`submitLeaguesFetch`) and projects the one requested league back out. The request/response shape was deliberately kept independent of this so a real single-league endpoint, if PoE ever adds one, can swap in behind `submitLeaguesFetch` with no wire-visible change — just a different `Exec` closure and a new dedup key scoped to one league instead of a whole realm/type
- [ ] Reusable rate-limited priority request queue (PoE OAuth API now, PoE Legacy API later): every PoE OAuth API request should flow through an internal, prioritized, de-duplicated queue rather than firing directly — per `_reference/poe-apis/poe-apis.md` §5.5's already-documented "Queue-Per-Policy Architecture" (one serial queue per rate-limit policy, dispatched one at a time with a minimum-safe delay computed from the live `X-Rate-Limit-*-State` headers per §5.6). **The generic scheduler core is built** (`internal/reqqueue`, 2026-07-13): priority-ordered dispatch with FIFO tie-break, `Submit`-time de-dup/merge-to-max-priority (multiple `Waiter`s fan out from one in-flight task), `SetPriority`/`Cancel` for an already-queued task, and per-policy state learned from a `HeaderParser` callback supplied per API — instantiated separately per API (one for PoE OAuth; a second, fully separate instance whenever the not-yet-built Legacy API needs one) rather than one manager internally juggling both, per an architecture review by Opus (2026-07-13): the two APIs hit different hosts with independent rate-limit budgets that never constrain each other, so unifying the managers would only add coupling — a shared telemetry sink, not a shared manager, is the answer if cross-API observability is ever wanted. Same review caught a real correctness bug during implementation: two same-policy tasks becoming ready at the same instant were both dispatched as concurrent goroutines, racing their actual completion order even though priority *selection* order was correct — fixed by tracking one in-flight dispatch per policy key and holding the next same-policy task back until it clears (poe-apis.md §5.5's "dispatched serially," not just "paced by a computed delay"), covered by `TestQueue_PriorityOrder_HighestDispatchesFirstUnderContention`. Blocking vs. non-blocking is `Waiter.Wait(ctx)`: a caller uninterested in blocking just discards the `Waiter` and lets the task's `Exec` side effects (cache write, topic publish) happen on their own; a caller that wants inline delivery calls `Wait` with a bounded ctx, and an expiring/cancelled ctx only abandons that one wait — the task keeps running regardless (covered by `TestWaiter_Wait_ContextTimeout_TaskStillCompletesInBackground`), consistent with the review's finding that a WS `wait: true` request must add a waiter goroutine on top of the same async dispatch rather than blocking the per-connection read loop inline (see `server.go`'s serial read loop comment). **The PoE OAuth adapter is now wired up too** (2026-07-13): `poeOAuthRateLimitHeaders` (`internal/server/poe_ratelimit.go`) parses the named-policy headers per §5.1 (including multi-tier rules — a single rule name's limit/state values are comma-separated lists, one `reqqueue.Rule` per tier); `s.poeQueue` (`server.go`) is a `*reqqueue.Queue` instance constructed alongside `s.poeClient`, sharing `s.rootCtx`; and the `/profile` fetch (`internal/server/poe_profile.go`) is the first real caller, submitting through `s.poeQueue` with a stable `PolicyHint` (`poeOAuthProfilePolicyHint`) and per-request `priority`/`wait` params exposed over `poe.profile.locale`/`poe.profile.twitch` (see the PoE OAuth data endpoints entry above). The Legacy API's own adapter and queue instance are still not built (no Legacy API data-fetching exists yet at all). Two documented simplifications remain in the core itself: no proactive HEAD-request policy discovery (§5.4) — a task with no known policy dispatches ungated the first time, same as poe-apis.md describes for a never-before-seen endpoint; and the computed delay for a saturated rule is that rule's full `Period` (a safe but not maximally efficient upper bound), not §5.6's exact "time until this rolling window resets," which would need a request-timestamp history this package doesn't keep. One thing worth remembering when testing against this: the real header parser reads period/restriction as whole seconds, so the smallest meaningful gate constructible in a test is 1s period + the 1s buffer = 2s — give any such test's timeouts comfortable headroom above that, not just barely over it (`TestEnsurePoeProfile_PriorityAffectsDispatchOrderUnderContention` needed a fix for exactly this). **`Queue.Policies()` added** (2026-07-13): a sorted, point-in-time snapshot of every policy the queue currently knows about, for `poe.ratelimit.status`/`TopicPoeRateLimit` to expose live budget without duplicating `reqqueue`'s internal state into every domain payload — reading it requires care about *when*: `dispatch` only updates a policy's stored state *after* a Task's `Exec` returns (from that response's headers), so anything wanting post-fetch state (like the rate-limit-status push) must wait on the same `Waiter` the fetch itself returns rather than reading `Policies()` from inside `Exec`, which would only ever see the prior fetch's state (`internal/server/poe_ratelimit.go`'s `publishPoeRateLimitStatusAfter` exists specifically for this sequencing)
- [ ] poe-info-service tunable-constants file (cache TTLs, queue/rate-limit settings): a new file (e.g. `internal/server/poe_constants.go`) colocating every cache-TTL/queue/rate-limit tunable this work and future work introduces (default per-field profile cache ages, the minimum enforced re-fetch age floor, queue worker settings) so they're easy to see in aggregate and adjust — distinct from `internal/poe/oauth.go`'s existing fixed protocol constants (`ClientID`, `Scopes`, `AuthURL`/`TokenURL`, `CallbackPath`), which stay put since they're facts about the protocol, not tunable knobs
- [ ] Multi-account PoE OAuth support: the `accounts` table records the local player's own PoE OAuth identity on login/refresh/logout (`poe_uuid`, `oauth_credential_key`, `oauth_authenticated_at` — see `poe_oauth.go`'s `upsertOAuthAccount`/`clearOAuthAccountActive`, schema v10), merged onto the same row `Client.txt` guild events already populate by `name`. But the credential itself is still a single global token under one fixed key (`poeOAuthToken`, ADR-005) — "one active OAuth identity at a time" per ADR-005. Recommended design (an architecture review by Opus, 2026-07-13, weighed concurrent multi-login against switching): **switching, not concurrency** — keep one live token/refresh-timer at a time, but store each known account's token under a `sub`-parameterized credential key (`poeOAuthToken:{sub}`) instead of the bare constant, so "switch account" becomes load-a-different-stored-token + re-arm refresh + flip `accounts.oauth_authenticated_at`, never running concurrent refresh loops for the rare multi-account case. `oauth_credential_key` already exists on the `accounts` row to record which key an account's token lives under — today it's always the bare constant; this item is what makes it actually per-account
- [ ] Account identity: key `accounts` by both `name` and `poe_uuid`; reconcile renames on OAuth login: `accounts` is currently unique only on `name`, and `upsertOAuthAccount`'s `ON CONFLICT(name)` means a PoE account rename (display name changes, `poe_uuid`/`sub` stays the same) inserts a second row rather than updating the existing one — splitting guild-event history from OAuth identity across two rows for the same real account. Fix: also make `poe_uuid` a unique key; OAuth login resolves identity by `poe_uuid` first, and if a row with that `poe_uuid` already exists under a different `name`, updates that row's `name` to the newly returned username rather than inserting a new row. `name` remains its own unique key and the sole identity for accounts known only from `Client.txt` guild events (a friend's account, say) that have never gone through an OAuth login locally and so have no `poe_uuid` at all
- [ ] Account selector (name or uuid) on account-attributable WebSocket requests: not every known account will have a `poe_uuid` — an account seen only via `Client.txt` chat/whisper/guild events (e.g. a friend's account, never OAuth-authenticated on this machine) has no `sub` claim to key on, only a `name`. **Built for `poe.profile.locale`/`poe.profile.twitch`** (2026-07-13, `resolvePoeAccount` in `internal/server/poe_profile.go`): an optional `account` param, either a `poe_uuid` or a plain `name`, resolves to whichever `accounts` row matches; empty defaults to the currently OAuth-active account. A selector naming a real but not-currently-active account resolves successfully but with no access token (this service holds only one live OAuth credential at a time, ADR-005) — such a request can still be served from cache, just never freshly fetched, until multi-account credential switching exists (see "Multi-account PoE OAuth support"). Still needed: the same pattern on chat/whisper/session queries, and on characters/stash once those OAuth data endpoints exist
- [ ] "Scan filesystem for install directories" button on the Game settings page: invokes an on-demand filesystem scan RPC on poe-info-service (Steam library folders, Program Files, GOG, etc.) to find PoE installs, independent of whether the game process is currently running — distinct from the process-based auto-detect (`internal/detect`) that watches for a running game process
- [ ] Steam presence: richer retry/backoff for outbound Steam requests: the initial implementation (`internal/steam`) only retries transport-level errors a couple of times with a short fixed backoff; no exponential backoff, jitter, or circuit-breaking after sustained failure
- [ ] Steam presence: resolve Steam vanity URLs (custom profile names) to steamid64 server-side, so a user doesn't have to look up their raw 64-bit Steam ID to populate `steam_id`
- [ ] Steam presence: Steam OpenID webview login (distinct from the Steam Web API key, which has no OAuth/programmatic issuance path at all — see ADR-007) to auto-fill the user's own steamid64 into `steam_id`, the same way l2p-poe's WebView already captures `POESESSID`
- [ ] Steam presence: distinguish a private Steam profile from "not currently playing anything" — the miniprofile scrape currently gives no reliable signal to tell these apart, so both surface as an identical, empty rich-presence entry today
- [ ] Steam presence: support tracking friends (additional steamids) for group play — `steam_id` was deliberately narrowed to a single value (this project supports only one local Steam-based PoE client today, see CONTRIBUTING.md's "Steam presence" section); tracking a friends list again would mean going back to a list-shaped config/API, this time built for that purpose from the start rather than retrofitted
- [ ] Steam presence: investigate rich presence as a primary league/level/zone source for detecting game state on a different computer (e.g. a Steam Deck) than the one running poe-info-service — Client.txt isn't locally readable in that scenario, but the same league/level parsed from rich presence today (`character.level`/`character.class`/`poe.league`, see CONTRIBUTING.md) could stand in for what Client.txt normally provides, if fetched from wherever poe-info-service actually runs

## Goal: Chat
- [ ] Chats tab — channel-number filtering: the Filter panel UI is built but "show only global #3" / "show only trade #2" can't be wired up until `chats` has a `channel_number INTEGER` column (schema migration to v4) and poe-info-service's ingest writer (`poe-info-service/internal/ingest/writer.go`) tracks the current channel join per install so new rows get the right number on ingest
- [ ] Copy support for chat/DM excerpts: select one or more message rows in the chat or DM view and copy them as plain text so conversations can be shared on forums or Discord without combing the raw log
- [ ] Local chat capture: parse and store local (area) chat lines from `Client.txt` so the Local checkbox in the chat filter panel becomes functional; requires identifying the log line format and adding a `local` channel variant to the ingest worker
- [ ] DM/whisper push notification while tabbed out: fire a system-tray or OS notification when an incoming whisper arrives and the game does not hold focus; hooks into the live event bus whisper emission so no separate polling is needed
- [ ] Tab-out chat client: compose and send a single message to the game's chat box via keystroke injection while the player is out-of-game; one typed message → one keystroke sequence delivered to the client's input box → one in-game send; limited to one message at a time per ADR-004 (one outside action maps to one inside action); depends on the game window being open and the player being logged in
- [ ] Optional unified chat view across concurrently-logged-in accounts: chat/whisper capture is inherently per-install/session (the `Client.txt` tailer), so a user running two PoE clients on two accounts at once naturally gets two separate per-account chat views today. Default should stay per-account rather than auto-merging — unlike Stash (see the Stash screen entry), merging two accounts' chat into one stream is lossy: you lose which identity a given whisper was even to, which can be actively confusing (two different people whispering "invite?" look identical once merged). Offer a unified cross-account stream only as an explicit opt-in for the rare user who wants a single firehose across both sessions
